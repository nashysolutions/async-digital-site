# Async Digital — Brand

A one-page index to every brand asset in this repo. Read this first. Follow the links for detail.

The goal: a non-designer can produce a leaflet, business card, or email signature using only what this page points to, with no calls to Rob or Helen.

## The foundations

Four files under `brand/` hold the decisions. Everything else is applied work.

- **Colour**: [`brand/colour.md`](brand/colour.md). Two brand colours, warm neutrals, three status colours, contrast ratings for every pairing used on the site.
- **Typography**: [`brand/typography.md`](brand/typography.md). System stack on the web, Inter in print, Arial in email. One type scale across all three.
- **Voice and tone**: [`brand/voice.md`](brand/voice.md). How Async Digital writes. Prefer, avoid, and when in doubt.
- **Positioning**: [`brand/positioning.md`](brand/positioning.md). How Audient sits next to Speed to Lead. What gets mentioned where, what doesn't.

If you're editing copy, start at voice — then check positioning if Audient might come up. If you're placing colour or type, start at colour or typography.

## Logo

### About the mark

Six horizontal bars of varying widths, each the same height, each starting at a different horizontal offset. They represent **asynchronous threads of work running in parallel** — the company name in visual form. Each bar is a thread; the bar's width is the size of the work on it; the varied starts are threads kicking off at different times. The irregularity is the point, not a stylistic accident. A future redesign should preserve the "parallel threads of irregular duration" reading; a sorted or monotonic bar pattern is the failure mode.

The mark is a single-colour form, filled with the brand `--primary` (brick) by default. It accepts any `color:` override via CSS `currentColor`, so dark-surface and monochrome print use is a one-property swap.

### Lockups

All files live under [`brand/logo/`](brand/logo/). Each lockup is a self-contained SVG with transparent background.

| File                    | Use                                                                              |
| ----------------------- | -------------------------------------------------------------------------------- |
| `symbol.svg`            | Mark only. Square. Avatars, app icons above favicon size, tight spaces.          |
| `horizontal.svg`        | Mark + two-line wordmark (Async / Digital). Default for web header and letterhead.|
| `stacked.svg`           | Mark above single-line "Async Digital". Vertical compositions, pitch-deck title slides. |
| `wordmark.svg`          | Outlined "Async Digital" without the mark. Rare — email headers, footer credits. |
| `favicon-mark.svg`      | Simplified three-bar variant for favicon and app-icon use at ≤48px.              |
| `clear-space.svg`       | Diagram showing the minimum padding rule (see below).                            |

### Monochrome variants

For single-ink print runs and dark surfaces, use the hardcoded mono variants rather than overriding colours in-place:

- `symbol-mono-black.svg`, `horizontal-mono-black.svg`, `stacked-mono-black.svg`
- `symbol-mono-white.svg`, `horizontal-mono-white.svg`, `stacked-mono-white.svg`

### Export bundle

Raster and PDF exports live in [`brand/logo/exports/`](brand/logo/exports/). Regenerated by `scripts/build-logo.py`.

- `{symbol,horizontal,stacked}-{1x,2x,3x}.png` — 512 / 1024 / 1536 px wide for web fallback, presentations, and email.
- `{symbol,horizontal,stacked}.pdf` — vector PDF for print suppliers.
- `favicon-{16,32,48}.png`, `favicon.ico`, `apple-touch-icon.png` — site favicon set, generated from `favicon-mark.svg` for legibility at small sizes.

### Clear-space rule

The mark needs a minimum clear-space of **one bar height** around all four sides (15% of the mark's width, 15% of its height). No text, image, or page edge should sit closer. See `brand/logo/clear-space.svg` for the diagram.

### Minimum size

- Horizontal lockup: not below 120px wide (web) or 25mm wide (print).
- Stacked lockup: not below 80px wide.
- Symbol: not below 32px square. Below that, use `favicon-mark.svg`.

### Typeface

The wordmark is **Inter Bold** outlined to paths and embedded in every SVG, so the files render identically everywhere without needing Inter installed on the viewer's machine. The font source is bundled at [`brand/logo/fonts/`](brand/logo/fonts/) under SIL OFL 1.1.

### Regenerating

```
uv run scripts/build-logo.py
```

This rebuilds every SVG, PNG, PDF, and favicon from scratch, and syncs the site-facing copies into `assets/`.

## Print templates

All print sources are SVG with text outlined from Inter Bold / Inter Regular, so the files are self-contained and don't depend on the recipient having Inter installed. Dimensions are authored in millimetres. Print-ready PDFs are regenerated by `scripts/build-cards.py`.

### Business cards ([`brand/print/`](brand/print/))

| File                                     | Use                                                      |
| ---------------------------------------- | -------------------------------------------------------- |
| `business-card-helen-front.svg`          | Helen's card — front (name, GDC, email, phone).          |
| `business-card-back.svg`                 | Shared back — symbol only, centred, brick on warm bg.    |
| `business-card-template-front.svg`       | Blank front for future hires; placeholders in `[brackets]`. |
| `exports/*.pdf`                          | Print-ready PDFs for the supplier. RGB colour space.     |
| `exports/*.png`                          | Low-fi previews for reviewing layout without opening a PDF. |

### Card spec

- **Trim size**: 85×55mm (UK standard).
- **Bleed**: 3mm every side → PDF canvas is 91×61mm.
- **Crop marks**: rendered on every PDF at the four trim corners (1mm clear of the trim, 3mm long, 0.2mm stroke).
- **Colour space**: RGB. Moo converts to CMYK on their side. The warm-bg / brick palette is low-risk for RGB→CMYK shift, but always **test-print a single card before ordering the full 250** — ICC profile and stock interact.
- **Safe zone**: keep critical content ≥3mm inside the trim on every side.

### Supplier

- **Moo Originals** (or the Classic range) — decided 2026-04-17.
- Lead time: ~5–7 working days UK.
- Budget: £20–50 for 250 cards, double-sided.
- Upload: `business-card-helen-front.pdf` as side 1, `business-card-back.pdf` as side 2. Upload at "press-ready with bleed" and let Moo auto-detect the trim.

### Regenerating

```
uv run scripts/build-cards.py
```

Edit the `HELEN` or `TEMPLATE` dictionaries at the top of `scripts/build-cards.py` to change content. Layout sizes (cap heights, margins, baseline offsets) live in the same file — tune, re-run, compare PNG previews, then send the PDF to Moo.

### Adding a new hire's card

1. In `scripts/build-cards.py`, duplicate the `HELEN = {...}` block with the new person's details.
2. Add a call in `main()`: `svgs["business-card-<name>-front"] = front_svg(bold, NEW_PERSON)`.
3. Re-run the script. Ship the new PDF to Moo.

## Folder layout

How `brand/` is organised as content lands.

```
brand/
  colour.md          — palette tokens, contrast, don'ts
  typography.md      — typefaces, type scale, licences
  voice.md           — voice, tone, vocabulary
  logo/
    *.svg            — lockups, monochrome variants, wordmark, clear-space
    exports/         — PNG @1x/2x/3x, PDF, favicons
    fonts/           — Inter TTFs (SIL OFL 1.1) bundled for print / email
  print/
    business-card-*.svg — card artwork (front per person, shared back, blank template)
    exports/         — print-ready PDFs + PNG previews
  email/             — HTML signatures, Mailchimp master
  slides/            — Gamma / Keynote masters
```

The three foundation files stay flat inside `brand/`. Applied assets (logo, print, email, slides) live in their own subfolders so each set of templates can be zipped and handed to a printer or mail client without picking through the rest.

## How brand tokens reach live surfaces

Two files translate the foundations into code that the site uses at runtime.

- **`assets/brand.css`** ([`#13`](https://github.com/async-digital-ltd/async-digital-site/issues/13)): the colour and type tokens from `brand/colour.md` and `brand/typography.md` as CSS custom properties. Every page in this repo, and every demo page at `demo.async-digital.com`, consumes the same file over HTTPS.
- **`assets/brand.js`** ([`#11`](https://github.com/async-digital-ltd/async-digital-site/issues/11)): shared constants the markup can't easily hold. Currently just the contact email, wired via `data-brand-mailto`.

If a brand value changes, the update order is:

1. The note in the knowledge vault (`projects/incubating/async-digital-brand-foundation.md`). That's the decision log.
2. The relevant file under `brand/` here. That's the documentation.
3. `assets/brand.css` (or `assets/brand.js`). That's the code downstream consumers read.

Changing step 3 without steps 1 and 2 leaves the documentation lying. Changing steps 1 and 2 without step 3 leaves the live site wrong.

## Versioning and breakage

`brand.css` and `brand.js` are loaded directly from `https://async-digital.com/assets/` by the main site, the demo site, and any future property. There's no semver; every consumer gets the current file.

Two implications:

- **Renaming a CSS custom property is a breaking change.** Search the org for `var(--old-name)` before renaming, and update every consumer in the same PR.
- **Adding a property is safe.** New tokens can land here ahead of their use.

If a consumer ever needs to pin a specific version (e.g. a print supplier's template reads `brand.css` at generation time), cut a tag and let them load from the tag's raw URL rather than `main`. That case hasn't arisen yet.

## Where the decisions live

This repo holds the documentation and the code. The reasoning behind each decision (why warm brick over teal, why a system stack over a webfont, why no em dashes) lives in the knowledge vault under `projects/incubating/async-digital-brand-foundation.md`. If a value needs to change, the vault note changes first.
